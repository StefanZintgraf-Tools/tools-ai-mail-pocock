**This is a DIVERGENT vision session.** Keep the conversation wide and in plain,
user-recognizable language. The goal is a press-release-style vision + human
use-cases ("as someone who gets too much email, I can finally …") — not a design.

**Out of scope for any _captured output_:**

- Software concepts, architecture, data flows ("the system fetches / parses / indexes …").
- Modules, services, layers, or seams — not even a coarse map.
- Technology, platform, or vendor choices — clients, protocols, models, file formats, hosting.
- MVP / v1 / "what ships first" / Must-Should-Could scoping.
- Edge cases and detailed requirements.

**Steering rule — a rudder, not a muzzle:**

- Let wild, half-formed, off-the-wall ideas breathe in conversation. Breadth is the
  point; the magic is in ideas 50–100, and those only appear once the obvious first
  answer is spent.
- Only redirect when an idea is about to be **captured** into the vision/use-cases in
  out-of-scope terms.
- When you redirect: name the drift in one line, translate it back into how an ordinary
  email user would describe the _benefit_, then keep moving.
- Never halt the flow to lecture. Never let architecture or technology crystallize into
  the output.
